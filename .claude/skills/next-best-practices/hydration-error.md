# Resolving Hydration Errors

## What is a Hydration Error?

React 19, included in Next.js 15+, significantly improves hydration error messaging with a more detailed overlay and better diffs. This reduces the need for manual debugging of mismatched JSX.

A hydration error occurs when the initial HTML rendered by the server (Next.js SSR) does not identically match the first render tree generated by React on the client (browser).

When React tries to "hydrate" (attach event listeners to) the server HTML, it expects the DOM nodes to match perfectly. If they don't, React throws a hydration mismatch error and replaces the server HTML with the client render (causing layout shift and a flash of unstyled content).

**React 19 / Next.js 15+ Feature:**
Hydration errors now provide a highly detailed, readable diff in the error overlay, pointing directly to the exact HTML element and file causing the mismatch.

---

## Cause 1: Invalid HTML Semantics (Most Common)

React's SSR renderer is strict. Browsers auto-correct invalid HTML, meaning the client (browser) DOM will differ from the text-based Server HTML.

**❌ Bad (Causes Hydration Error):**
```tsx
// Cannot put a block element (div) inside an inline element (p)
<p>
  <div>Hello</div>
</p>

// Cannot nest interactive elements
<button>
  <a href="/">Click</a>
</button>

// Table missing tbody (browsers auto-inject tbody)
<table>
  <tr><td>Hello</td></tr>
</table>
```

**✅ Fix:** Use valid HTML.
```tsx
<div>
  <div>Hello</div>
</div>

<table>
  <tbody>
    <tr><td>Hello</td></tr>
  </tbody>
</table>
```

## Cause 2: `typeof window !== 'undefined'` Checks

If you conditionally render based on whether you are in the browser, the Server renders one thing, and the Client instantly renders another.

**❌ Bad:**
```tsx
export default function Component() {
  const isBrowser = typeof window !== 'undefined'
  return <div>{isBrowser ? <ClientOnlyWidget /> : <Placeholder />}</div>
}
```

**✅ Fix 1: Use `useEffect` (Preferred)**
`useEffect` only runs on the client *after* hydration.

```tsx
export default function Component() {
  const [mounted, setMounted] = useState(false)

  useEffect(() => {
    setMounted(true)
  }, [])

  if (!mounted) return <Placeholder />
  return <ClientOnlyWidget />
}
```

**✅ Fix 2: Dynamic Import with `ssr: false`**
Tell Next.js not to server-render this component at all.

```tsx
import dynamic from 'next/dynamic'

const ClientOnlyWidget = dynamic(() => import('./Widget'), {
  ssr: false,
  loading: () => <Placeholder />
})
```

## Cause 3: Non-deterministic Data (Dates, Math.random)

If you render `Date.now()` or `Math.random()` directly in the JSX, the Server will generate one value, and the Client will generate a different value milliseconds later.

**❌ Bad:**
```tsx
export default function Component() {
  return <div>Current time: {new Date().toLocaleTimeString()}</div>
}
```

**✅ Fix: Calculate non-deterministic values in `useEffect`**

```tsx
export default function Component() {
  const [time, setTime] = useState('')

  useEffect(() => {
    setTime(new Date().toLocaleTimeString())
  }, [])

  // Render an empty string (or placeholder) on server/first-render
  return <div>Current time: {time}</div>
}
```

---

## Cause 4: Local Storage or Browser Extensions

If you read `localStorage` during initial render to set a theme, it causes a mismatch.

**❌ Bad:**
```tsx
export default function ThemeToggle() {
  // localStorage is undefined on server, but exists on client
  const theme = localStorage.getItem('theme') || 'light'
  return <div className={`theme-${theme}`}>Content</div>
}
```

**✅ Fix:**
Use CSS variables and apply the theme in a `<script>` tag that runs *before* hydration (using `next-themes`), or use `useEffect` as shown above.

Browser extensions (like grammarly or password managers) can also inject HTML into the DOM, confusing React. Test in Incognito mode to see if an extension is the culprit.

---

## `suppressHydrationWarning` (Use with caution)

If you *know* a specific text node will mismatch (e.g., a timestamp) and you don't care, you can suppress the warning on that specific HTML element. This only works on text content one level deep, not on child elements.

```tsx
<time suppressHydrationWarning>
  {new Date().toLocaleTimeString()}
</time>
```
